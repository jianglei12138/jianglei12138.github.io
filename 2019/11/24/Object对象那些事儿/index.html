上篇我们提到，对Java对象加载创建过程中Object#<clinit>是第一个调用的clinit方法，那这次我们就来分析一下Java对象的顶级父类对象Object。Object类位于java.lang包中，java.lang包包含着Java最基础和核心的类，在编译时会自动导入。每个类都使用 Object 作为超类。所有对象（包括数组）都实现这个类的方法。可以使用类型为Object的变量指向任意类型的对象

Object类分析

Object整个源码特别简单，主要是一系列的native方法，除去一些不重要的重载方法外，Object主要有以下几个方法

registerNatives

根据OpenJDK源码在 jdk/src/share/native/java/lang/Oject.c中我们可以找到对应的实现

其目的就是通过Jni静态注册的方式去动态注册Object类中native方法的实现

hashCode

我们首先来说一下Object#hashCode的实现，可能会有人觉的hashCode就是Object对应的内存地址经过一系列的转换得到，从jvm的定义角度来说 “hashCode来源于这个对象的内部地址转换成的整型值”，这个角度是正确的，但是从具体实现来说这种说法不一定是完全正确的，我们来从JVM_IHashCode的实现分析HashCode究竟是怎么计算出来的，JVM_IHashCode实现位于OpenJDK的hotspot/src/share/vm/prims/jvm.cpp

可见调用的是ObjectSynchronizer::FastHashCode静态方法实现，由于当对象初始化的时候是没有生成hashCode，只有当应用程序显示的调用时才会创建，第一次创建完成后，便会缓存在对象头中， 下一次就可以直接获取，保证效率同时也保证对同一个Object对象hashCode的唯一性。

可见真正生成HashCode的方法是get_next_hash，关于get_next_hash实现实际上是有多重不同的实现方案，通过编译器的一个参数指定，参数就叫hashCode

  • hashCode == 0，使用os::random生成随机数，随机数的生成策略是一种线性取余方式生成的
  • hashCode == 1，采用把obj地址对象转成int，通过异或等操作实现计算
  • hashCode == 2，返回固定值1，用来做测试
  • hashCode == 3，返回一个自增的序列
  • hashCode == 4，返回对象地址
  • 其他，也是现在java默认的hashCode的实现，采用的是线程的指定状态 + Marsaglia’s xor-shift的随机数生成算法

另外java中存在几种特殊的hashCode实现，基本数据类型对应的对象，数组以及String

数组也是一个普通的java对象，因此计算hashCode的方式与Object相同，但是这种方式的弊端是当我修改的数组中的内容，数组的hashCode不会发生任何变化，因此推荐使用Arrays#hashCode方法计算一维数组的hashCode，对于多维度数组可以通过Arrays#deepHashCode实现

String类因为复写了equals方法，比较的是String内部char数组的每一位是否相同，因此hashCode实现也与char数组相关，要保证两个equals的String，hashCode必须相等。

getClass

getClass返回的是运行时对象的class，那这个运行时类是什么？通过阅读jdk源码我们来一探究竟，由Oject.c可知，getClass实现

如果对象为空，抛出空指针异常，否则直接调用GetObjectClass获取到对应的jclass对象，这里就和我们jni开发过程中使用的一类接口是一样的了，但是这个GetObjectClass返回的是什么呢？具体实现位于openjdk\hotspot\src\share\vm\prims\jni.cpp中

所以总结可知，GetObjectClass返回的是kclass的_java_mirror，具体即java/lang/Class实例，关于kclass与oop我们可以单独写一遍文章做介绍

equals

equals方法完全是由java实现的，纯粹是对比两个对象内存地址是否一致

此处有一个疑问，equal与hashCode究竟有什么关系？参考Object注释我们可以知道，对于hashCode而言

  1. 每一个调用hashCode方法，返回值应该保持一致
  2. 如果两个对象equal相等，则hashCode应该一致
  3. 当两个对象equal不相等，不要求hashCode一致

即如果我们重写了equal方法。则一定要重写hashCode，如果不重写会产生什么问题？

因为hashCode比较常用的使用地方是map我们可以通过map去分析，如果我们使用了自定义equals方法的对象作为map的key，当我们插入是会先通过hash函数找到存放的数组索引，对于两个equals相同的对象，我们期望的是他们的索引应该是相同的，但是如果我们没有修改hashCode方法，这里就可能得到两个不同的索引值，无法满足想要实现的效果

clone

顾名思义，clone方法是复制，克隆的意思，就是复制一份相同的对象，clone方法默认是protect的，我们无法直接访问，如果需要clone则需要实现 Cloneable 接口，并把clone方法Override成public，如果我们不实现 Cloneable 直接Override会怎么样？JDK注释中已经明确告我们如果这样子做会抛出CloneNotSupportedException异常，为什么呢？我们通过源码去找答案

上述提到jdk只是提供了浅拷贝，那么什么是浅拷贝，什么是深拷贝呢? 不难想到如果按照这种实现方式我们是无法实现对局部变量对象的拷贝的，即虽然我们创建了两个不同的对象，但是他们属性对象指向的仍然是同一个，那应该怎么实现深拷贝？

  1. 如果对象属性只是包含基本数据类型，String，以及其数组等，浅&深拷贝相同
  2. 属性对象存在普通对象时，可以重写普通对象的clone方法实现新的对象创建，在当前对象clone方法中调用其clone方法，或者直接在clone方法中new一个新的对象，并且把对象的属性通过新创建的方式赋值
  3. 如果可以直接把对象实现序列化接口，写入在读出即新的对象

wait/notify/notifyAll

这三个方法与线程息息相关，我们放在一起研究下，通过阅读相关注释我们可以知道，在三个函数使用前必须先获得锁，一般配合synchronized 关键字使用，一般在synchronized同步代码块里使用。当线程执行wait()方法时候，会释放当前的锁，然后让出CPU，进入等待状态。只有当 notify/notifyAll() 被执行时候，才会唤醒一个/多个正处于等待状态的线程，然后继续往下执行，直到执行完synchronized 代码块的代码或是中途遇到wait() ，再次释放锁。

可见实现都是交给ObjectSynchronizer，具体我们等待分析线程相关内容的时候，做统一了解

finalize

finalize是一个空实现的java方法，当垃圾回收器将要回收对象所占内存之前被调用，即当一个对象被虚拟机宣告死亡时会先调用它finalize()方法，让此对象处理它生前的最后事情（这个对象可以趁这个时机挣脱死亡的命运）

finalize()的调用具有不确定行，只保证方法会调用，但不保证方法里的任务会被执行完。
